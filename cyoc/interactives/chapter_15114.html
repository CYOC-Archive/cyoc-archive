    She couldn't think of any problems in her profile changes so she clicked the 'CONTINUE'
    button. What she didn't realize was that her children's profiles were directly linked to her
    own so when ‘CONTINUE’ was clicked, the Chronivac found two link conflicts. It quickly
    scanned the two conflicting subject profiles and found that both subjects were currently
    inanimate objects. Since user authorization was given to complete any profile changes, the
    Chronivac used a directive loophole regarding general inanimate objects.
    A minor programming glitch allowed the Chronivac to use it’s unprotected 'from inanimate
    to inanimate' transformation protocols regarding the inanimate objects that Jessica and Jeff
    currently were.
    The Chronivac’s default inanimate-to-inanimate transformation protocols quickly found a few
    discrepancies in the two conflicting profiles and did what any good program does with a
    corrupt data file. Both subject profiles were quickly rebuilt into simpler object profiles.

    Up to this point, said objects have received no additional changes per se. Jeff was currently
    still enjoying his time in support of his mother and Jessica was happily looking out into an
    empty bedroom from her box in her mother‘s doll display case.
    The only problem just changing their profiles caused was to remove all possibility of any
    end user of the Chronivac emitter and program from ever restoring them. In such a case, they
    would have to ship the objects to Chronivac Corp where the facilities exist to build new
    subject profiles for said objects.
    The only problem was, the Chronivac program wasn't able to stop there. It had found more
    then one discrepancy in each of the two object profiles. The profile rebuilds had eliminated
    all but one of the discrepancies.
    This discrepancy, however, was in the percent of transformation field.
    In any living subject transformation, this data field holds a numerical representation of how
    complete the transformation is to be. The coded number answers questions such as:
  • Will whatever remain conscious?
  • Will whatever have access to memories?
  • Will whatever be allowed any movement?
  • Will whatever have any characteristics of the original subject?
    • Now; the simple object profile uses this same field as a percent of transformation completed
    so the transformation could be stepped for the user's watching pleasure or as an indicator of
    an interrupted transformation sequence.
    Because of the drastic size difference of the field, the data was truncated to fit. It would be
    pure random chance if the field held the '100' necessary to prevent the Chronivac from
    restarting. Since the program's logic tree requires the completion of every transformation
    sequence, any simple object which has an object profile that shows that it’s morphing is
    still incomplete immediately is completed to match what is called out by said program's
    standard object template library. It's subject profile rework would then be allowed to continue
    by possibly deleting any unnecessary linked object profiles.
    Just so you know; removing profiles from disk storage does not remove anything from existence.
    The Chronivac program just wont recognize said person or object as being transformed.
